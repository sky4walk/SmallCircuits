Was ist das?

Diese Seite simuliert die Programmausführung eines HT46F47E-basierten TPS-Programmierbausteins, so wie er in eigenen Steuerungsprojekten per Tastendruck programmiert wird (drei Taster S1/S2/S3 am Chip, siehe sendCommandsToTPS() in der Python-Referenz). Statt echte Tastenfolgen an Hardware zu senden, liest dieser Simulator ein .tps-Programm ein, führt es Befehl für Befehl in Software aus und zeigt Register, Ein-/Ausgänge und PWM live auf einem virtuellen Front-Panel an.

Alles läuft vollständig lokal im Browser — es werden keine Daten irgendwohin gesendet und keine Verbindung zu einem Server aufgebaut.

Taster-Programmierung erklärt

Der HT46F47E-Baustein wird auf echter Hardware ohne PC programmiert — nur über drei Taster S1, S2, S3. Das bildet sendCommandsToTPS() und sendDigit() aus dem Python-Referenzskript nach:

  • S3 (Modus): aus dem Ruhezustand einmal gedrückt fordert den Programmiermodus an; ein zweites Mal während der Eingabe beendet ihn wieder (Handshake, analog zum S2+S3-Ablauf in sendCommandsToTPS()).
  • S1 (Zählen): jede Ziffer (0–F) wird eingegeben, indem S1 entsprechend oft gedrückt wird — der Zähler läuft nach 15 wieder auf 0 um.
  • S2 (Bestätigen): übernimmt den aktuellen Zählerstand als Ziffer. Zwei Ziffern ergeben einen vollständigen Befehl (erst die Befehls-Ziffer, dann die Parameter-Ziffer) und damit eine fertige Programmadresse — genau wie im Original, wo sendDigit() zweimal pro Eintrag in commands aufgerufen wird.

Fertig erfasste Adressen erscheinen rechts in der Tabelle und lassen sich per In Editor übernehmen direkt in den normalen Programm-Editor oben übertragen und dort weiter simulieren.

Die Sequenz-Vorschau geht den umgekehrten Weg: Sie nimmt das bereits im Editor assemblierte Programm und spielt automatisch genau die Tastenfolge ab, die man am echten Gerät eingeben müsste — inklusive Log jeder einzelnen Ziffer. Die realen Wartezeiten aus sendDigit() (0,5 s pro Tastendruck) sind hier per Tempo-Regler frei einstellbar, da sie in Software keine elektrische Bedeutung haben.

Bedienung

1. Programm laden: Entweder direkt im Editor tippen/einfügen, per .tps laden… eine Datei von der Festplatte öffnen, oder mit Beispiel: test.tps das mitgelieferte Beispiel laden.

2. Assemblieren: Prüft das Programm auf Formatfehler und erzeugt die Disassemblierungs-Tabelle links unten. Fehlerhafte Zeilen werden übersprungen und rot als Warnung angezeigt (wie im Original-Parser).

3. Ausführen: Einzelschritt führt genau einen Befehl aus. Run lässt das Programm automatisch laufen (Tempo per Schieberegler einstellbar) — bei Befehl 2 ("Wartezeit") wird zusätzlich kurz pausiert, proportional zum Parameter, damit sich der Effekt sichtbar macht. Reset setzt CPU und Register zurück auf die erste Programmadresse.

4. Ein-/Ausgänge beobachten und setzen: Das DIP-18-Panel zeigt die Pin-Namen des HT46F47E. Die vier LEDs rechts (Dout1–4) spiegeln die tatsächlichen Ausgänge. Links können die vier Eingänge E1–E4 per Kippschalter gesetzt werden, AD1/AD2 per Schieberegler (0–15) und S1/S2 per Taster — das Programm reagiert live darauf, z. B. bei bedingten Sprüngen (Befehl 12).

Dateiformat (.tps)

Jede Zeile beschreibt genau einen Programmschritt:

ADR HH #Kommentar

  • ADR — 3-stellige Dezimaladresse (000–127), entspricht der Zeilennummer im Programm.
  • HH — 2 Hex-Ziffern: die erste ist der Befehl (0–14, siehe Tabelle unten), die zweite der Parameter (0–15).
  • #Kommentar — optional, wird ignoriert, dient nur der Lesbarkeit.

Leere Zeilen und Zeilen, die mit # beginnen, werden übersprungen. Beispiel:

000 13 #Ausgaenge PA4 bis PA7 ein.

→ Adresse 0, Befehl 1 (Portausgabe), Parameter 3.

Unterschiede zur Python-Referenz (tps.py)

Beim Nachbauen wurden folgende Stellen bewusst anders umgesetzt als im ursprünglichen Skript — zur Kontrolle hier transparent aufgelistet:

  • Befehl 6 (Parameter 4–8, "A = Din…") und Befehl 12 (Parameter 4–11, "wenn Din.x…"): Im Original wird hier versehentlich auf die Ausgangs-Pins (Dout1–4) statt auf die Eingangs-Pins (E1–E4) zugegriffen (Copy&Paste-Fehler, teils zusätzlich mit doppeltem Attributzugriff cpuDef.cpuDef…, was in Python zum Absturz führen würde). Dieser Simulator wertet stattdessen tatsächlich die Eingänge E1–E4 aus, wie es die Spezifikations-Kommentare im Skript vorsehen.
  • Befehl 0: Im Original wird der Programmzähler nicht erhöht, was auf echter Hardware vermutlich zu einem Zustand führt, der so nie beabsichtigt war. Der Simulator hält die Ausführung an dieser Stelle sauber an, statt in eine unsichtbare Endlosschleife auf derselben Adresse zu laufen.
  • Befehl 7, Parameter 6 (A = A / B): Ganzzahlige Division (abgerundet Richtung 0) statt einer Gleitkommazahl, plus Abfangen von Division durch 0 (statt eines Python-Fehlers).
  • Registerbreite: Wie im Original werden A/B/C/D nicht automatisch auf 4 Bit begrenzt — Addition/Subtraktion kann rechnerisch über 15 oder unter 0 gehen. Das entspricht 1:1 dem Referenzverhalten.

Falls eines dieser Verhalten für ein konkretes Projekt anders erwartet wird, einfach Bescheid geben — lässt sich gezielt anpassen.
