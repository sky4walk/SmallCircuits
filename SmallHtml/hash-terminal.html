§1 Was macht MD5 eigentlich?

MD5 (Message Digest 5, 1992 von Ron Rivest entworfen) nimmt eine beliebig lange Eingabe entgegen und erzeugt daraus einen 128-Bit (16 Byte, 32 Hex-Zeichen) langen "Fingerabdruck". Gleiche Eingabe → immer gleicher Hash. Schon ein einziges verändertes Bit in der Eingabe wirbelt den kompletten Ausgabewert durcheinander (Lawineneffekt).

Intern arbeitet MD5 mit vier 32-Bit-Registern A, B, C, D, die mit festen Startwerten initialisiert werden und sich Block für Block verändern, bis am Ende der finale Hash aus ihnen zusammengesetzt wird.

§2 Schritt 1 — Padding (Auffüllen der Nachricht)

MD5 verarbeitet die Nachricht in 512-Bit-Blöcken (64 Byte). Damit die Länge passt, wird die Nachricht so aufgefüllt:

  • Ein einzelnes Bit 1 wird angehängt (als Byte 0x80).
  • Danach folgen so viele 0-Bytes, bis die Gesamtlänge ≡ 56 (mod 64) ist — also noch genau 8 Byte bis zum vollen Block fehlen.
  • In diesen letzten 8 Byte wird die ursprüngliche Länge der Nachricht in Bit als 64-Bit-Zahl im Little-Endian-Format gespeichert.

Das garantiert: jede gepolsterte Nachricht ist ein exaktes Vielfaches von 64 Byte — und die Länge steckt trotzdem verschlüsselt im letzten Block, was Längenverlängerungs-Tricks erschwert.

§3 Schritt 2 — Verarbeitung in 512-Bit-Blöcken

Jeder 64-Byte-Block wird in 16 Wörter à 32 Bit zerlegt (M[0]…M[15], Little-Endian gelesen). Die aktuellen Registerwerte A,B,C,D werden in Arbeitsvariablen a,b,c,d kopiert, dann laufen 64 Runden in vier Gruppen zu je 16 ab. Am Ende jedes Blocks werden a,b,c,d zu den laufenden Registern A,B,C,D addiert (mod 2³²) — genau wie ein Fahrradtacho, der bei Überlauf einfach wieder bei 0 anfängt.

§4 Schritt 3 — Die vier Rundenfunktionen

Jede der vier 16er-Gruppen benutzt eine andere nichtlineare Funktion auf b, c, d, um die Bits gründlich zu vermischen:

  • F(b,c,d) = (b∧c) ∨ (¬b∧d) — "wenn b, nimm c, sonst d" (Runde 1)
  • G(b,c,d) = (b∧d) ∨ (c∧¬d) — "wenn d, nimm b, sonst c" (Runde 2)
  • H(b,c,d) = b⊕c⊕d — reines XOR, Paritätsprüfung (Runde 3)
  • I(b,c,d) = c⊕(b∨¬d) (Runde 4)

In jeder Runde wird zusätzlich addiert: die passende Konstante K[i], ein Wort M[g] aus dem Nachrichtenblock (die Reihenfolge g variiert pro Runde) und das Ergebnis wird anschließend um eine feste Anzahl Bit nach links rotiert (<<<). Rotation statt Verschiebung ist wichtig — dabei geht kein einziges Bit verloren.

§5 Die Konstanten K[0..63]

Die 64 Konstanten stammen nicht aus dem Nichts, sondern werden aus der Sinusfunktion abgeleitet: K[i] = ⌊ |sin(i+1)| · 2³² ⌋. Das liefert 64 scheinbar zufällige, aber jederzeit reproduzierbare 32-Bit-Zahlen — praktisch, denn niemand muss eine Tabelle "erfinden" oder verstecken; jeder kann sie selbst nachrechnen.

§6 Ist MD5 heute noch sicher?

Nein. MD5 gilt seit rund 2005 als kryptografisch gebrochen: Kollisionen (zwei verschiedene Eingaben mit demselben Hash) lassen sich in Sekunden auf gewöhnlicher Hardware erzeugen. Für Signaturen, Zertifikate oder Passwort-Hashing ist MD5 ungeeignet. Für Prüfsummen von Dateien (z. B. "ist der Download unbeschädigt angekommen?") wird es aus Kompatibilitätsgründen aber immer noch verwendet — dort geht es nicht um Angriffsresistenz, sondern nur um zufällige Übertragungsfehler.

§7 Nachvollziehbares Beispiel: "abc"

Die Nachricht abc (3 Byte) wird auf 64 Byte gepolstert: Byte 0x80, 52-mal 0x00, dann die Länge 24 Bit als 8-Byte Little-Endian-Zahl.

§1 Was macht SHA-256?

SHA-256 (Secure Hash Algorithm, Teil der SHA-2-Familie, NSA/NIST 2001) erzeugt aus beliebig langer Eingabe einen 256-Bit (32 Byte, 64 Hex-Zeichen) Hash. Es ist der aktuelle Industriestandard — u. a. in Bitcoin, TLS-Zertifikaten und Git-Objekten (seit Git 2.29 optional) im Einsatz.

Statt vier Registern wie MD5 nutzt SHA-256 acht 32-Bit-Register a…h, initialisiert mit den ersten 32 Bit der Nachkommastellen der Quadratwurzeln der ersten acht Primzahlen (2, 3, 5, 7, 11, 13, 17, 19).

§2 Schritt 1 — Padding

Fast identisch zu MD5: Anhängen von 0x80, dann Nullen bis Länge ≡ 56 (mod 64), dann die Original-Länge in Bit in den letzten 8 Byte. Unterschied: SHA-256 speichert die Länge Big-Endian (höchstwertiges Byte zuerst), MD5 dagegen Little-Endian.

§3 Schritt 2 — Message Schedule (16 → 64 Wörter)

Jeder 64-Byte-Block liefert zunächst 16 Wörter w[0..15]. SHA-256 erweitert diese auf 64 Wörter, indem für i = 16…63 aus bereits berechneten Wörtern neue gemischt werden:

s0 = (w[i-15] ⋙ 7) ⊕ (w[i-15] ⋙ 18) ⊕ (w[i-15] ≫ 3)
s1 = (w[i-2] ⋙ 17) ⊕ (w[i-2] ⋙ 19) ⊕ (w[i-2] ≫ 10)
w[i] = w[i-16] + s0 + w[i-7] + s1 (mod 2³²)

⋙ = Rotation (Bits wandern rüber), ≫ = normale Verschiebung (Bits fallen raus). Diese Mischung sorgt dafür, dass jedes der 64 Wörter von vielen ursprünglichen Bytes abhängt.

§4 Schritt 3 — 64 Kompressionsrunden

Pro Runde i werden zwei "Summen" S0, S1 aus Rotationen von a bzw. e gebildet, dazu zwei Auswahlfunktionen:

  • Ch(e,f,g) = (e∧f) ⊕ (¬e∧g) — "e entscheidet, ob f oder g gilt" (Choose)
  • Maj(a,b,c) = (a∧b) ⊕ (a∧c) ⊕ (b∧c) — Mehrheitsentscheid über 3 Bit (Majority)

Daraus zwei Zwischenwerte:

temp1 = h + S1 + Ch(e,f,g) + K[i] + w[i]
temp2 = S0 + Maj(a,b,c)

Danach rutschen alle acht Register eine Position weiter (h←g, g←f, f←e, e←d+temp1, d←c, c←b, b←a, a←temp1+temp2) — ähnlich einer Schieberegister-Kette. Nach 64 Runden werden a…h zu den laufenden Hash-Registern addiert.

§5 Die Konstanten K[0..63]

Auch hier keine "Zauberzahlen": Die 64 Konstanten sind die ersten 32 Bit der Nachkommastellen der Kubikwurzeln der ersten 64 Primzahlen (2, 3, 5, 7, 11, …, 311). Wie bei MD5's Sinus-Trick dient das dazu, jeden Verdacht auf absichtlich eingebaute Hintertüren ("nothing up my sleeve numbers") auszuräumen.

§6 MD5 vs. SHA-256 auf einen Blick
  • Ausgabelänge: MD5 = 128 Bit, SHA-256 = 256 Bit
  • Register: MD5 = 4 × 32 Bit, SHA-256 = 8 × 32 Bit
  • Runden pro Block: beide 64, aber SHA-256 mit deutlich mehr Durchmischung (Message Schedule) davor
  • Sicherheit: MD5 gebrochen (Kollisionen praktisch erzeugbar), SHA-256 gilt Stand heute als sicher
  • Typischer Einsatz heute: MD5 nur noch für unkritische Prüfsummen, SHA-256 für Signaturen, Zertifikate, Blockchains, Git
§7 Nachvollziehbares Beispiel: "Hallo Welt"
