Notes on this port ▶

This page is a direct port of my2DMatrix.java, QLearning.java and ReinforcementLearning_Hanoi.java to JavaScript. The 27-state reward matrix (setRMatrix()) and the episode/update logic are carried over value-for-value, including the original (int) truncation on the Q-update.

The 27 ASCII-art states from setMovesVisible() were decoded into actual disk/peg positions (small, medium, large → peg A/B/C) so the towers above render real graphics instead of text art — the state numbering itself is unchanged.

Two original quirks were kept as-is rather than “fixed”, since the goal was a faithful conversion: the traceRoute loop condition uses || instead of &&, and getBestNextState falls back to state 0 when no action has a positive Q-value yet (which can show up as an odd jump early in training, before enough episodes have run). A hard iteration cap was added purely as a browser safety net against an infinite loop — happy to tidy either up if you'd rather have the “correct” version.

The disk count is generalized beyond the original 3-disk program: at 3 disks the page still uses the exact hand-built reward matrix and state numbering from setRMatrix(); for any other disk count the reward matrix and state numbering are generated from the same legal-move rules. Because the original Q-update overwrites instead of blending (no learning rate) and truncates to an integer, a reward of 100 decays below 1 — and gets crushed to 0 — after roughly log(100)/log(1/λ) hops from the goal. At λ = 0.5 that's about 6 hops, which happens to just cover the 7-move 3-disk solution but leaves more distant states permanently at Q = 0 once you add disks. Raise λ toward 0.9–0.95 for 4+ disks so values can reach further back through the state graph; even then, more disks means a much larger state space and a far less likely random walk to the goal each episode, so convergence gets slow and unreliable rather quickly. That's a property of this particular algorithm, not a bug in the port.
